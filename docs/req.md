# Требования к проекту "Restaurant Advisor Bot"

## 1. Введение

Restaurant Advisor Bot - это Telegram-бот, помогающий пользователям находить рестораны с учетом персональных предпочтений и текущего местоположения. Бот использует внешние API для получения данных о заведениях и LLM-модель для генерации персонализированных объяснений рекомендаций.

## 2. Функциональные требования

### 2.1. Общие для всех пользователей

#### 2.1.1. Авторизация
- **Через Telegram**: пользователь автоматически распознается по Telegram ID при взаимодействии с ботом. Первое обращение создает учетную запись с ролью `User`.
- **Через HTTP**: доступ к REST-эндпоинтам, защищенным API-ключом.

#### 2.1.2. Роли
- **User** - обычный пользователь бота.
- **Admin** - администратор.

#### 2.1.3. Healthcheck
- **GET /healthcheck** - публичный эндпоинт, возвращающий:
  - `status`: `"UP"` или `"DOWN"`;
  - `authors`: список ФИО.

#### 2.1.4. Пользователи (только для Admin)
- **GET /users** - возвращает список всех зарегистрированных пользователей (Telegram ID, роль, дата регистрации).

#### 2.1.5. Валидация ввода
- **Название города** в команде `/location set <город>` проверяется регулярным выражением `^[a-zA-Zа-яА-ЯёЁ\s-]{2,50}$`. При несоответствии бот отвечает: *"Укажите корректное название города (только буквы, пробелы и дефис)"*.
- **Текстовый запрос** в `/search <текст>` ограничен 100 символами. При превышении бот предлагает сократить запрос.
- **Команды с параметрами** при неверном формате, например, `/preferences set` без текста, выводят подсказку с синтаксисом: *"Использование: /preferences set <ваше предпочтение>"*.
- **Невалидные типы сообщений**: при получении от пользователя любого сообщения, не являющегося текстовой командой или ожидаемой геолокацией (например, изображение, видео, аудио, файл, стикер), бот отвечает: *"Пожалуйста, используйте текстовые команды или отправьте геолокацию. Введите /help для списка доступных команд."*

### 2.2. Управление предпочтениями

Пользователь может задать свои диетические ограничения и вкусовые предпочтения.

- **Изменение предпочтений**  
  Команда: `/preferences set <текст>`  
  Примеры: `/preferences set вегетарианец`, `/preferences set аллергия на шашлык`.  
  **Ответ**:  
  - При успехе: *"Предпочтения обновлены: <новый список предпочтений>"* (список выводится в виде нумерованного перечня).  
  - При превышении лимита (более 10 записей): *"Достигнут лимит предпочтений (10). Удалите ненужные через /preferences remove <номер>"*.

- **Удаление предпочтений**  
  Команда: `/preferences remove <номер>`  
  Пример: `/preferences remove 3`  
  **Ответ**:  
  - При успехе: *"Предпочтение '<удаленный текст>' удалено."*  
  - При указании несуществующего номера: *"Предпочтение с таким номером не найдено. Проверьте список через /preferences show"*.  
  - Если список предпочтений пуст: *"У вас нет сохраненных предпочтений для удаления."*

- **Просмотр текущих предпочтений**  
  Команда: `/preferences show`  
  **Ответ**: нумерованный список всех установленных предпочтений. Если список пуст: *"У вас пока нет сохраненных предпочтений. Добавьте их с помощью /preferences set <текст>"*.


### 2.3. Управление местоположением

- **Установка местоположения**  
  Команда: `/location set <город>` или отправка геолокации (Telegram location message).  
  **Ответ**:  
  - При успешной установке по городу: *"Местоположение сохранено: <город> (центр города)"*.  
  - При отправке геолокации: *"Местоположение обновлено по вашей геопозиции"*.  
  - При ошибке валидации города: *"Укажите корректное название города (только буквы, пробелы и дефис)"*.

- **Просмотр текущего местоположения**  
  Команда: `/location show`  
  **Ответ**: *"Текущее местоположение: <город или координаты>"*. Если местоположение не задано: *"Местоположение не указано. Используйте /location set <город> или отправьте геолокацию."*

### 2.4. Поиск ресторанов

- **Поиск по запросу**  
  Команда: `/search <кухня/ключевые слова>`  
  Пример: `/search суши`, `/search итальянский ресторан с видом`.  
  **Ответ**: при успешном поиске выводится сообщение со списком найденных заведений (до 5 результатов). Для каждого ресторана показывается карточка, содержащая:
  - Название
  - Адрес
  - Рейтинг (если доступен)
  - Кнопки: **"Почему мне это подходит?"** и **"Добавить в список для посещения"**.  
  Если ничего не найдено: *"По вашему запросу ничего не найдено. Попробуйте изменить параметры."*  
  Если поиск уже выполняется: *"Дождитесь завершения текущего поиска"*.  
  При отсутствии параметров поиска используются сохраненные предпочтения; если и они не заданы, бот отвечает: *"Уточните запрос или задайте предпочтения через /preferences set"*.

- **Случайный ресторан рядом**  
  Команда: `/random`  
  **Ответ**: одна карточка ресторана в радиусе ~2 км от текущего местоположения. Если поблизости ничего нет: *"Рядом не найдено ресторанов. Попробуйте изменить местоположение."*

- **Объяснение рекомендации**  
  После команды `/search` или `/random` бот выводит карточку ресторана с кнопкой **"Почему мне это подходит?"**. При нажатии кнопки:
  - Если объяснение еще не сгенерировано, бот отправляет промежуточное сообщение *"Генерирую объяснение…"*.
  - После получения ответа от LLM или по таймауту пользователь получает сообщение с текстом объяснения либо сообщение об ошибке: *"Не удалось сгенерировать объяснение. Попробуйте позже"*.

### 2.5. Список "Посетить"

- **Добавление в список**  
  Для каждого найденного ресторана выводится информационная карточка с кнопкой **"Добавить в список для посещения"**. При нажатии:  
  **Ответ**: *"Ресторан '<название>' добавлен в ваш список."*  
  При достижении лимита (50): *"Достигнут лимит списка. Удалите ненужные записи или отметьте их как посещенные"*.

- **Просмотр списка**  
  Команда: `/visit list`  
  **Ответ**: нумерованный перечень всех ресторанов в списке. Для каждого указывается:
  - ID записи
  - Название
  - Адрес (если сохранен)
  - Статус (не посещен / посещен)  
  Если список пуст: *"Ваш список для посещения пуст."*

- **Отметка о посещении**  
  Команда: `/visit mark <id ресторана>`  
  **Ответ**: *"Ресторан '<название>' отмечен как посещенный и перемещен в историю."*  
  При неверном ID: *"Запись с таким ID не найдена. Проверьте список через /visit list."*

- **Удаление из списка**  
  Команда: `/visit remove <id ресторана>`  
  **Ответ**: *"Ресторан '<название>' удален из списка."*  
  При неверном ID: *"Запись с таким ID не найдена."*

- **История посещений**  
  Команда: `/history`  
  **Ответ**: перечень ресторанов, отмеченных как посещенные. Для каждого выводится:
  - Название ресторана
  - Дата добавления в историю
  - Адрес, кухня (если сохранены)  
  Список ограничен последними **50** записями.  
  Если история пуста: *"История посещений пока пуста"*.

### 2.6. Обработка длительных операций

#### 2.6.1. Статусы запросов
- Каждый поиск ресторанов (`/search`, `/random`) получает уникальный `requestId`.
- Пользователь может проверить состояние запроса командой:  
  `/status <requestId>`  
  Возможные статусы:
  - `Выполняется` - запрос обрабатывается.
  - `Готово` - результат получен, ответ отправлен.
  - `Ошибка` - что-то пошло не так, попробуйте позже.
- Пока запрос в статусе `Выполняется`, повторный запуск поиска тем же пользователем блокируется ботом с сообщением: *"Дождитесь завершения текущего поиска"*.

#### 2.6.2. Таймауты и повторные попытки
- **Google Places API**: при отсутствии ответа за 5 секунд выполняется одна повторная попытка через 2 секунды.
- **GigaChat**: при отсутствии ответа за 30 секунд пользователь получает сообщение: *"Не удалось сгенерировать объяснение. Попробуйте позже"*. Повторных попыток для LLM не предусмотрено.
- При исчерпании квот любого из API бот сообщает: *"Сервис временно недоступен. Попробуйте позже"*.

#### 2.6.3. Защита от повторной генерации объяснения
- При нажатии кнопки **"Почему мне это подходит?"** бот проверяет, не выполняется ли уже генерация объяснения для того же ресторана и того же пользователя.
- Если активный запрос существует, бот отвечает сообщением:  
  *"Объяснение уже генерируется, пожалуйста, подождите"*.  
  Новый запрос к LLM не инициируется.
- Состояние отслеживается по связке `(telegramId, restaurantId)` и сбрасывается после завершения генерации.

### 2.7. Администрирование (только для роли Admin)

Администраторы имеют доступ к следующим командам в боте:

- `/list-users` - показывает список всех зарегистрированных пользователей. Вывод ограничен последними 50 записями.
- `/add-admin <telegramId>` - назначает пользователя администратором.
- `/remove-admin <telegramId>` - понижает администратора до обычного пользователя. Нельзя снять права с последнего администратора.
- `/list-admins` - выводит список всех администраторов.

## 3. Нефункциональные требования

### 3.1. Производительность
- Время ответа на команду поиска ресторанов без учета LLM-генерации и без учета внешних API не должно превышать 3 секунд в 95% случаев.
- LLM-объяснение может занимать до 30 секунд. Пользователь получает промежуточное сообщение "Генерирую объяснение…".

### 3.2. Масштабируемость
- Приложение рассчитано на одновременную работу до 20 активных пользователей.
- Данные пользователей, предпочтений, местоположений, списков "Посетить" и истории посещений хранятся в СУБД **PostgreSQL**.  
- Для локальной разработки и тестирования используется контейнеризированный PostgreSQL, запускаемый вместе с приложением через `docker compose`.

### 3.3. Безопасность
- Внешние API‑ключи (Telegram, Google Places, GigaChat) хранятся в переменных окружения, не попадают в репозиторий.
- Административные HTTP‑эндпоинты защищены ключом.

### 3.4. Удобство использования
- Все команды бота имеют краткое описание, доступное через `/help`.
- При неверном вводе бот предлагает подсказку с корректным форматом команды.

## 4. Допущения и ограничения

- **Внешние API**  
  - Для данных о ресторанах используется Google Places API.  
  - Для генерации объяснений используется GigaChat Lite.  
  - В случае исчерпания квот API‑запросов бот возвращает сообщение об ошибке с предложением повторить позже.

- **Хранение данных**  
  - В качестве основной СУБД применяется **PostgreSQL**. Для упрощения развертывания в репозитории предоставляется файл `docker-compose.yml`, поднимающий контейнер с PostgreSQL и приложением.

- **Геолокация**  
  - Если пользователь не отправил геолокацию и не указал город, поиск выполняется по городу по умолчанию: "Санкт-Петербург".  
  - Поиск "рядом" при отсутствии точных координат использует центр указанного города.

- **LLM‑объяснения**  
  - Генерация объяснения вызывается явно пользователем, чтобы избежать лишней нагрузки на API и задержек при простом поиске.

- **Очередь сообщений**  
  - В текущем варианте использование очереди сообщений не требуется, так как нагрузка предполагается низкой и все операции выполняются синхронно.

- **Лимиты на пользовательские данные**  
  - Максимальное количество ресторанов в списке "Посетить": **50**. При попытке добавить 51-й бот отвечает: *"Достигнут лимит списка. Удалите ненужные записи или отметьте их как посещенные"*.  
  - Максимальное количество сохраненных предпочтений: **10**.

## 5. Технический стек

- **Язык программирования**: Java 25
- **Фреймворк**: Spring Framework 7
- **Сборка**: Gradle
- **База данных**: PostgreSQL
- **Контейнеризация**: Docker, Docker Compose
- **Публикация образа**: образ должен быть загружен на Docker Hub
- **Telegram API**: используется для приема команд и отправки ответов
- **LLM API**: GigaChat
- **Картографические данные**: Google Places API
